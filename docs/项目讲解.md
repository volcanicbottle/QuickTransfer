# 多端互通 — 项目讲解

> 这是一个**局域网内的"文件传输助手"**:在每台电脑上跑一个小 C++ 服务,浏览器打开就能在
> 电脑之间、电脑和手机之间互发文字和文件。不需要装客户端,手机用浏览器扫同一局域网的
> 地址即可使用。

---

## 1. 一张图看懂整体结构

```
        ┌─────────────────────── 电脑 A ───────────────────────┐
        │  duoduan (C++ 进程)                                   │
        │  ┌──────────┐  ┌─────────┐  ┌─────────┐  ┌────────┐  │
        │  │ httplib  │  │Discovery│  │ History │  │ Auth   │  │
        │  │ HTTP服务 │  │UDP 广播 │  │ SQLite  │  │ SQLite │  │
        │  └────┬─────┘  └────┬────┘  └─────────┘  └────────┘  │
        │       │             │            ▲ EventBus(SSE推送) │
        └───────┼─────────────┼────────────┼────────────────────┘
   静态页面+API │        UDP 38520         │
                │             │            │
   ┌────────────▼──┐   ┌──────▼───────┐   ┌▼──────────────┐
   │ 本机浏览器     │   │ 电脑 B 的     │   │ 手机浏览器     │
   │ localhost:8520│   │ duoduan 进程  │   │ 192.168.x.x:  │
   │ (完全信任)    │   │ (/peer/* 互发)│   │ 8520 (token)  │
   └───────────────┘   └──────────────┘   └───────────────┘
```

三种"客户端"访问同一个 HTTP 服务,信任级别不同:

| 来源 | 走哪些接口 | 怎么证明身份 |
|---|---|---|
| 本机浏览器 (127.0.0.1) | `/api/*` 全部 | 不用证明,localhost 全放行 |
| 局域网里另一台电脑 | `/peer/text` `/peer/file` `/peer/pair` | 请求头 `X-Pair-Token`(配对时换到的共享密钥) |
| 手机浏览器 | 少量 `/api/*` 白名单 | Cookie `qt_token`(首次输 PIN 注册时发的) |

## 2. 目录结构

```

  server.{h,cpp}   App 类:HTTP 路由主体(消息、文件、SSE)

  auth.{h,cpp}     AuthStore(SQLite 存配对密钥和手机 token)+ PinGuard(PIN 防爆破)
  discovery.{h,cpp} UDP 广播发现局域网里其他 duoduan 实例
  history.{h,cpp}  消息历史,SQLite
  events.{h,cpp}   EventBus:进程内发布-订阅,服务端事件推到浏览器(SSE)
  util.{h,cpp}     小工具:时间戳、url 编码、随机密钥、唯一文件名……

  index.html / app.js / style.css   前端,由 C++ 进程直接托管

  httplib.h(HTTP库) sqlite3(数据库) json.hpp(JSON) doctest.h(测试)
tests/             doctest 单元测试
```

## 3. 启动流程(main.cpp → App::run)

1. `main.cpp:10` 数据目录默认 `~/.duoduan`,支持 `--data-dir/--port/--name/--download-dir` 覆盖。
2. `Config::load` 读 `config.json`;第一次运行会生成随机设备 id 和 **6 位 PIN** 并保存。
3. `App::run()`(`src/server.cpp:17`):
   - 创建下载目录和暂存目录(`staging/`,发送文件前先存这里);
   - 从 8520 开始**最多试 10 个端口**绑定(`server.cpp:23`),都被占就报错退出;
   - 启动 `Discovery`:每 3 秒往 UDP 38520 广播一次"我是谁、在哪个端口",同时监听别人的广播,
     维护一张在线设备表(10 秒没声音算离线);
   - `setup_routes()` 注册所有 HTTP 路由,然后 `listen_after_bind()` 阻塞跑服务。

## 4. HTTP 接口一览

### 给本机浏览器用的(`/api/*`,localhost 免鉴权)

| 接口 | 作用 |
|---|---|
| `GET /api/self` | 我是谁。localhost 访问额外返回 PIN(给界面显示) |
| `GET /api/peers` | 设备列表:UDP 发现的电脑 + 注册过的手机,带在线/已配对状态 |
| `GET /api/events` | **SSE 长连接**,服务端实时推送新消息/进度/设备变化 |
| `GET /api/messages?peer=` | 和某设备的历史消息 |
| `POST /api/send-text` | 发文字给某设备 |
| `POST /api/send-file?peer=` | 上传文件到暂存区,后台线程推给目标电脑 |
| `POST /api/retry` | 重发失败的消息 |
| `POST /api/pair` | 输入对方 PIN,向对方发起配对(仅 localhost) |
| `GET /api/file?id=` | 下载某条消息的附件 |

### 电脑与电脑之间互调的(`/peer/*`,凭 `X-Pair-Token`)

| 接口 | 作用 |
|---|---|
| `POST /peer/pair` | 对方拿着我的 PIN 来换共享密钥(配对) |
| `POST /peer/text` | 对方推文字过来 |
| `POST /peer/file` | 对方推文件过来(流式写 `.part`,完整后改名) |

### 手机浏览器用的(`/api/phone/*`,凭 Cookie)

| 接口 | 作用 |
|---|---|
| `POST /api/phone/register` | 首次:设备名 + PIN → 发一年期 `qt_token` Cookie |
| `POST /api/phone/send-text` / `send-file` | 手机发内容给这台电脑 |
| `POST /api/phone/heartbeat` | 每 10 秒心跳,维持"在线"状态 |

## 5. 一条消息的完整旅程

**电脑 A 给电脑 B 发文件:**

1. 浏览器把文件 POST 到 A 的 `/api/send-file?peer=B`;
2. A 先把文件完整收进**暂存区** `~/.duoduan/staging/`(`server.cpp:266`),写一条
   `status=pending` 的历史记录,立刻返回——浏览器不用等传输完成;
3. A 起一个**后台线程** `start_file_send`(`server.cpp:368`),拿出和 B 配对时存下的密钥,
   带上 `X-Pair-Token` 头,把文件流式 POST 到 B 的 `/peer/file`;
   每 200ms 通过 EventBus 发一次进度事件,浏览器的进度条就是这么动的;
4. B 校验 token → 流式写 `xxx.part` → 核对文件大小 → 改名成正式文件(防止读到半截文件),
   写入历史并推送 SSE,B 的浏览器立刻弹出新消息;
5. A 收到 200:删暂存文件、把消息标成 `ok`;失败则标 `fail`,暂存保留,等用户点"重试"。

**手机发文件给电脑**更简单:手机直接 POST 到电脑的 `/api/phone/send-file`,电脑存进下载目录,
没有第二跳。电脑发给手机时,文件留在暂存区,手机端界面出现"下载"按钮自己来拉
(`/api/file?id=`)——因为服务器没法主动往手机推。

## 6. 配对与鉴权(谁能跟谁说话)

- **电脑 ↔ 电脑**:A 的用户在界面输入 B 屏幕上显示的 PIN → A 调 B 的 `/peer/pair` →
  B 验证 PIN,生成一个随机**共享密钥 secret**,双方都存进各自 SQLite 的 `pairings` 表。
  以后每次互发消息都带 `X-Pair-Token: <secret>`,对得上才收。
- **手机 ↔ 电脑**:手机浏览器打开页面 → `/api/self` 返回 401 → 弹登录层 → 输设备名 + PIN →
  `/api/phone/register` 发一个一年有效的 `HttpOnly` Cookie。之后请求自动带 Cookie。
- **PIN 防爆破**(`auth.h:40` PinGuard):错一次后 1 秒内全部 429;连错 10 次锁 5 分钟。
  6 位数字有 100 万种组合,配合限速,暴力猜 PIN 不现实。
- **总闸门**:`set_pre_routing_handler`(`server.cpp:44`)在所有路由之前执行——
  非 localhost 的 `/api/*` 请求只放行 7 个白名单接口,且必须带有效手机 token,
  其余一律 403/401。这保证"危险接口"(比如让本机去给别人发消息的 `/api/send-text`)
  只有坐在这台电脑前的人能碰。

## 7. 前后端是怎么"结合"的

没有 Node、没有打包器、没有跨域:**C++ 进程本身就是 Web 服务器**。

- `svr_.set_mount_point("/", web_dir)`(`server.cpp:73`)把 `web/` 目录整个挂成静态站点,
  浏览器访问 `http://localhost:8520/` 拿到的 `index.html`、`app.js` 就是磁盘上那几个文件;
- 前端 `app.js` 用 `fetch()` 调同一地址下的 `/api/*`,因为页面和 API 同源,天然没有 CORS 问题;
- 服务端到浏览器的**实时推送**用 SSE(Server-Sent Events):前端
  `new EventSource('/api/events')`(`app.js:234`)挂一条长连接,后端任何线程调用
  `bus_.publish(...)`,所有订阅中的浏览器就会收到一条 JSON 事件(新消息、发送进度、设备上下线);
- 前端启动时调 `/api/self`,根据返回的 `is_remote` 决定走**桌面模式**(显示设备列表和 PIN)
  还是**手机模式**(单会话聊天界面),同一套 HTML/JS 服务两种终端(`app.js:271`)。

## 8. 几个工程细节(为什么这么写)

- **暂存区 + 立即返回**:上传和转发解耦,网页不会因为对方网速慢而卡住;失败可重试,
  因为暂存文件还在。
- **`.part` 后缀 + 改名**:接收方写完并核对大小后才 rename,避免下载目录里出现半截文件。
- **文件名只取末段** `fs::path(name).filename()`(`server.cpp:227`):防路径穿越攻击
  (对方传 `../../.bashrc` 这种名字)。
- **`util::unique_path`**:重名文件自动加序号,不覆盖已有文件。
- **EventBus 队列上限 1024**(`events.h:20`):浏览器断线不读时丢最旧事件,防内存涨爆。
- **SSE 5 秒保活**(`server.cpp:122`):没事件也发一行 `: ping` 注释,防止连接被中间设备掐断。
- **History 与 AuthStore 共用一个 SQLite 文件**,但各开各的连接,WAL 模式 + busy_timeout
  解决多线程并发写。

## 9. 安全模型一句话总结

> 局域网内、**应用层鉴权**(PIN 配对 → 共享密钥/Cookie)、**传输层明文**(HTTP 无加密)。
> 它能挡住"乱发请求的陌生设备",挡不住"在同一网络上抓包的人"。
> 这个权衡的详细讨论见《[问题解答](问题解答.md)》。
