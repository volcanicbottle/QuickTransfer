# 多端互通 —— 局域网文件传输工具 设计文档

日期：2026-06-10
状态：已确认

## 1. 目标

做一个类似微信"文件传输助手"的局域网互传工具：聊天式界面，可以发文件和文字消息，有历史记录。

- **第一阶段**：电脑 ↔ 电脑（Linux + Windows）
- **第二阶段**：手机（iOS/Android）通过浏览器接入，无需开发 App

## 2. 总体架构

每台电脑运行同一个 C++ 程序（单个可执行文件），它同时承担三个角色：

1. **HTTP 服务器**：给浏览器提供聊天界面和 API
   - 本机使用：浏览器打开 `http://localhost:8520`
   - 手机使用（第二阶段）：手机浏览器打开 `http://<电脑IP>:8520`
2. **HTTP 客户端**：把文件/消息推送给局域网内其他电脑的服务
3. **UDP 广播器**：周期性广播在线状态，实现设备自动发现

```
电脑A                                电脑B
┌─────────────────────┐            ┌─────────────────────┐
│ 浏览器(聊天界面)      │            │ 浏览器(聊天界面)      │
│   ↕ localhost:8520  │            │   ↕ localhost:8520  │
│ C++ 服务            │←─ HTTP ───→│ C++ 服务            │
│  · HTTP API + SSE   │←─ UDP广播 ─→│  (同一个程序)        │
│  · SQLite 历史记录   │            │                     │
└─────────────────────┘            └─────────────────────┘
```

## 3. 技术选型

| 部分 | 选择 | 理由 |
|---|---|---|
| HTTP 服务端+客户端 | cpp-httplib（header-only） | 一个头文件，跨 Linux/Windows，同时有 server 和 client |
| 浏览器实时推送 | SSE（Server-Sent Events） | cpp-httplib 原生支持（chunked response），比 WebSocket 少一个依赖；浏览器→服务端用普通 POST |
| 历史记录 | SQLite（C API） | 单文件数据库，零运维 |
| 前端 | 纯 HTML/CSS/JS 单页面，无框架 | 聊天气泡式界面，保持简单 |
| 构建 | CMake | Linux 用 g++，Windows 用 MinGW 或 MSVC |

## 4. 组件划分

- **discovery**：UDP 广播收发。广播包为 JSON：`{id, name, port}`，间隔 3 秒；维护在线设备表，10 秒未收到广播标记离线。
- **peer_api**：对其他电脑暴露的 HTTP 接口（接收文件 / 接收文本）。
- **local_api**：对本机浏览器暴露的 HTTP 接口（设备列表、历史记录、发送文件/文本、SSE 事件流）+ 静态页面。
- **history**：SQLite 读写。消息表字段：id、对端设备 id、方向（收/发）、类型（文本/文件）、内容（文本或文件路径）、文件大小、状态（成功/失败/进行中）、时间戳。
- **config**：设备 ID（首次启动生成 UUID 并持久化）、设备名（默认取主机名）、下载目录、端口。
- **web/**：前端静态文件（开发期从目录读取，便于迭代）。

## 5. 核心数据流

**发文件（A → B）**：
1. A 的浏览器把文件 POST 到本机服务（localhost，本地回环，速度极快）
2. A 的服务通过 HTTP POST 流式推送给 B 的服务（`peer_api`）
3. B 流式写入 `下载目录/xxx.part`，完成后改名为正式文件名
4. B 写入历史记录，通过 SSE 通知 B 的浏览器显示新消息
5. A 写入历史记录（状态：成功），SSE 更新 A 的界面和进度条

发送方副本策略：浏览器上传到 A 服务的文件先写入暂存目录；发送成功后删除暂存，历史记录只保留文件名和大小；发送失败时暂存保留，供"重试"使用（重试成功或用户删除该条消息后清理）。

**发文字**：同路径，payload 为文本，B 端浏览器可一键复制。

**设备发现**：服务启动后周期广播；前端通过 `local_api` 拉取在线设备列表并经 SSE 接收上下线事件。

## 6. 默认配置

- 端口：8520（被占用时自动尝试 8521、8522…）
- 下载目录：`~/Downloads/多端互通/`（Windows 为 `%USERPROFILE%\Downloads\多端互通\`）
- 设备名：默认主机名，可在设置中修改

## 7. 错误处理

- 对方离线或传输中断：消息标记"发送失败"，界面提供重试按钮；接收方删除不完整的 `.part` 文件
- 大文件：全程流式处理，不把整个文件读入内存
- 同名文件：自动改名（`文件名 (1).ext`）
- 端口冲突：自动递增端口；广播包里携带实际端口

## 8. 安全边界

第一版仅限可信局域网使用，不做身份验证——局域网内任何设备都可以发送文件和访问页面。配对 PIN 码列入第二阶段。

## 9. 范围外（明确不做）

- 文件夹传输（用户先手动压缩为 zip）
- 服务器中转/跨公网传输
- 剪贴板自动同步
- 原生手机 App

## 10. 测试策略

- 单元测试：history（SQLite 读写）、文件名冲突处理、广播包解析等纯逻辑
- 集成测试：同一台机器跑两个实例（不同端口、不同数据目录），验证发现与收发
- 手动验收：Linux ↔ Windows 真机互传大文件（>1GB）验证流式与进度

## 11. 里程碑

1. **M1**：服务骨架——HTTP 服务 + UDP 互相发现（页面上能看到对方设备）
2. **M2**：文字消息 + 聊天界面 + SSE 实时刷新
3. **M3**：文件传输（含进度条）+ SQLite 历史记录
4. **M4**：Windows 编译打包、收尾打磨
5. **第二阶段**：手机浏览器接入（手机↔电脑会话）、配对 PIN
