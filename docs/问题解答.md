# 问题解答(que.txt + server.cpp 里的注释提问)

整体讲解见《[项目讲解](项目讲解.md)》。这里逐条回答 que.txt 的问题,
以及你写在 `src/server.cpp` 注释里的疑问。

---

## que.txt 的三个问题

### 1)"明文 HTTP 太吓人了,不能防嗅探" — 对,确实不能

你的判断完全正确。**当前所有流量都是明文**:消息内容、文件内容、配对 PIN、
共享密钥 `X-Pair-Token`、手机的 `qt_token` Cookie,在网线/Wi-Fi 上都是裸奔的。
同一局域网里能抓包的人(Wireshark + ARP 欺骗,或开放 Wi-Fi 下直接监听)能看到一切。

当初这么设计的理由是威胁模型定为"家里/办公室的可信局域网",省掉了证书的麻烦。
如果要补上加密,可选方案从易到难:

| 方案 | 做法 | 代价 |
|---|---|---|
| **HTTPS(自签证书)** | httplib 编译时开 `CPPHTTPLIB_OPENSSL_SUPPORT`,用 `SSLServer`;首次启动生成自签证书 | 浏览器会弹"不安全"警告,手机上尤其烦;电脑互连时要 `enable_server_certificate_verification(false)` 或做证书指纹固定(pinning) |
| **证书指纹固定** | 配对时(PIN 已验证身份)顺便交换证书指纹,以后互连只认这个指纹 | 实现稍多,但体验最好:既加密又防中间人,浏览器警告只在首次出现 |
| **应用层加密** | 用 PIN/secret 派生密钥,对消息体做 AES-GCM | 不护住 URL/头部,浏览器端也要写一份加密逻辑,不推荐 |

推荐路线:**自签 HTTPS + 配对时交换证书指纹**。这是个独立的功能改造,可以列入下一步计划。

### 2)"为啥明文 HTTP 可以防止冒充?"

先纠正一个概念:**防冒充的不是 HTTP,是应用层的凭证**。HTTP 只是运输工具;
"防冒充"和"防嗅探"是两件独立的事:

- **防冒充(鉴权)**= 验证"你是谁"。本项目靠两样东西:
  - 电脑之间:配对时用 PIN 换来的**共享密钥**。每次发消息带 `X-Pair-Token` 头,
    接收方在 `check_pair_token`(`src/server_auth.cpp:22`)里和 SQLite 里存的比对,
    对不上就 403。陌生电脑没有这个密钥,所以**没法冒充已配对的设备**;
  - 手机:注册时发的 `qt_token` Cookie,每次请求在 `phone_from_request`
    (`src/server_auth.cpp:17`)里查表验证。
- **防嗅探(加密)**= 让偷看的人看不懂。这一层目前是**零**。

所以准确的说法是:这套机制能挡住"**只会发请求、不会抓包**"的攻击者——
比如局域网里某台中毒的机器拿扫描器乱打接口,会被 401/403 拦下。
但它挡不住能抓包的攻击者:**token 本身就在明文里飞**,抓到一次就能拿去冒充你。
换句话说,目前的"防冒充"建立在"没人嗅探"的假设上——这正是上一题要加 HTTPS 的原因:
加密不只是保护内容,也是在保护凭证,两个问题其实是同一个问题。

### 3)"前后端这里是怎么结合的?"

没有独立的前端服务器,**C++ 进程一人分饰两角**:

1. **静态托管**:`svr_.set_mount_point("/", web_dir)`(`src/server.cpp:73`)把 `web/`
   目录挂到根路径。浏览器打开 `http://localhost:8520/`,httplib 直接从磁盘读
   `index.html`、`app.js`、`style.css` 返回——和 nginx 托管静态文件一回事。
2. **API 调用**:页面里的 JS 用 `fetch('/api/...')` 调接口。页面和 API 是**同源**
   (同一个 host:port),所以没有任何 CORS/跨域配置。
3. **实时推送**:`new EventSource('/api/events')`(`web/app.js:234`)建立 SSE 长连接。
   后端把这条 HTTP 响应设置成 `text/event-stream` 并永远不结束,有事件就写一段
   `data: {...}\n\n`;前端 `es.onmessage` 收到后更新界面。这就是"对方发来消息,
   我的页面不刷新也能弹出来"的原理。
4. **同一套页面服务两种设备**:JS 启动时调 `/api/self`,后端根据来源 IP 返回
   `is_remote`;本机浏览器进桌面模式(设备列表 + PIN 展示),手机浏览器进手机模式
   (先弹 PIN 登录,然后是单会话聊天)。见 `web/app.js:271` 的启动自执行函数。

整个"前后端协议"就是:JSON over HTTP + SSE,接口清单见《项目讲解》第 4 节。

## server.cpp 注释里的疑问逐条解答

### `server.cpp:39` — "bus 是什么"

`bus_` 是 `EventBus`(`src/events.h`),一个进程内的**发布-订阅总线**,只有 30 行:

- 每个打开 `/api/events` 的浏览器对应一个 `Subscriber`(一把锁 + 一个条件变量 + 一个字符串队列);
- 任何业务代码调 `bus_.publish(json字符串)`,就把这条字符串塞进**所有**订阅者的队列,
  并唤醒它们;
- SSE 处理线程被唤醒后把队列里的内容写给浏览器。

作用:让"收到新消息"的线程和"挂着浏览器长连接"的线程解耦——发布方不需要知道
现在有几个浏览器开着。

### `server.cpp:45` — "`HandlerResponse::Unhandled` 这是什么"

`set_pre_routing_handler` 注册一个**在所有路由匹配之前执行**的钩子(httplib 提供的机制),
这里用它做全局鉴权闸门。钩子的返回值告诉 httplib 接下来怎么办:

- `Unhandled` = "我不管,**继续走正常路由**"(放行);
- `Handled` = "这个请求**我已经处理完了**(比如写了 403),别再往下走了"(拦截)。

所以 `if (is_local(req)) return Unhandled;` 的意思就是:本机来的请求,直接放行。

### `server.cpp:56` — "为什么这些可以 allow,这些是什么"

`kRemoteAllowed` 是**允许非本机设备(实际上就是手机)访问的 `/api/*` 白名单**:

| 接口 | 手机为什么需要它 |
|---|---|
| `/api/self` | 启动时问"我是谁/我是不是远程"(决定进手机模式) |
| `/api/events` | SSE 收新消息 |
| `/api/messages` | 拉自己会话的历史 |
| `/api/file` | 下载电脑发来的文件 |
| `/api/phone/send-text` / `send-file` | 手机发消息/传文件 |
| `/api/phone/heartbeat` | 报告在线 |

设计逻辑是**默认拒绝、按需放行**:凡是不在名单上的远程 `/api/*` 请求,有 token 也 403。
被排除在外的接口都是"指挥这台电脑做事"的:`/api/send-text`(让本机给别人发消息)、
`/api/pair`(让本机去配对)、`/api/retry`。这些只允许坐在电脑前的人(localhost)操作,
手机即使登录了也不该有这个权力。注意白名单只是第一道门——放行后每个接口内部
还会再验 token 并把数据范围限制到"这部手机自己的会话"。

### `server.cpp:68` — "`res.status = 401` 是怎么规定的"

这是 **HTTP 标准状态码**(RFC 9110 定义,不是本项目发明的),给 `res.status` 赋值
就决定了响应行里的数字。本项目用到的几个:

| 码 | 含义 | 本项目里什么时候出现 |
|---|---|---|
| 200 | 成功 | 默认 |
| 400 | 请求格式不对 | JSON 解析失败、缺字段 |
| 401 | **没带凭证/凭证无效**("你是谁?") | 手机没有有效 token |
| 403 | **身份明确但没权限**("我知道你是谁,但不行") | PIN 错误、pair token 不匹配、远程碰非白名单接口 |
| 429 | 请求太频繁 | PinGuard 节流/锁定中 |
| 502 | 上游(对方电脑)出错 | 配对时连不上对方 |

前端就靠这些数字分支提示语,比如 `app.js:46` 看到 429 显示"尝试次数过多"。

### `server.cpp:77` — "这个 this 是什么,lambda 吗"

是的,`[this](const httplib::Request& req, httplib::Response& res) { ... }` 是
**C++ lambda 表达式**,`svr_.Get("/api/self", lambda)` 把它注册为该路径的处理函数,
有请求进来时 httplib 会回调它。

方括号 `[this]` 是**捕获列表**:lambda 体里要用 `cfg_`、`port_` 这些 `App` 的成员变量,
而 lambda 本身是个独立的函数对象,默认看不见它们;捕获 `this`(App 对象的指针)之后,
`cfg_.id` 实际上就是 `this->cfg_.id`。可以理解为"把当前 App 对象借给这个回调用"。

(隐患提示:这要求 App 活得比服务器久——本项目里 App 在 `main` 栈上、`run()` 阻塞到
进程结束,所以安全。)

### `server.cpp:111` — `phone_from_request` "不懂"

这一行在 SSE 接口里判断:**远程来的连接必须是注册过的手机**。
`phone_from_request`(`src/server_auth.cpp:17`)做两步:

1. 从请求的 `Cookie` 头里取出 `qt_token` 的值(就是手机注册成功时服务器
   `Set-Cookie` 发下去的那串随机字符串,浏览器以后每次请求自动带上);
2. 拿它去 SQLite 的 `phones` 表查 `find_phone_by_token`——查到了,说明是注册过的手机,
   把手机信息填进 `ph` 返回 true;查不到返回 false,这里就给 401。

拿到 `ph.id` 之后还顺便做了**数据隔离**:`phone_filter = ph.id`,后面只把
"属于这部手机会话"的消息事件推给它,手机看不到电脑和其他设备的聊天。

### `server.cpp:114` — "`bus_.subscribe()` 这是什么"

给当前这条 SSE 连接在 EventBus 上**注册一个订阅者**,返回 `shared_ptr<Subscriber>`
(里面是空队列)。从此 `bus_.publish()` 的每条事件都会进这个队列。

接下来的 `set_chunked_content_provider` 是 httplib 的流式响应:httplib 反复调用
你给的 lambda 来要"下一块数据"。这个 lambda 在条件变量上**最多等 5 秒**——
队列有东西就打包成 `data: ...\n\n` 写出去;5 秒还没东西就写一行 `: ping\n\n`
(SSE 注释,浏览器忽略),目的是保活,防止连接被超时掐断。
最后一个参数(`server.cpp:144`)是连接断开时的清理回调,把订阅者从总线上摘掉,
不然断线的浏览器会留下一个永远没人消费的队列。
